# Text Selection Fix

## Problem Description

When users selected and copied text from PDFs generated by Fulgur, the entire paragraph or document would be copied instead of just the selected portion. This made text selection essentially unusable.

### Symptoms

- Selecting any character in a paragraph resulted in the entire paragraph being copied
- In some cases, selecting a small portion would copy megabytes of text from the document
- Text selection was broken for all content types (paragraphs, list markers, inline content)

### Root Cause

The PDF text selection mechanism relies on each glyph having an accurate `text_range` that indicates which portion of the original text that glyph represents. The Krilla PDF library uses these ranges to determine what text to select when a user makes a selection.

The bug was that **all glyphs in a paragraph were assigned the same text range: `0..text_len`**, where `text_len` was the length of the entire paragraph. This meant every glyph claimed to represent the entire text.

When a PDF reader processes text selection:
1. It maps the cursor position to a specific glyph
2. It uses that glyph's `text_range` to determine what text to select
3. Since all glyphs had `0..text_len`, selecting any glyph selected the entire paragraph

## Debugging Process

### Initial Investigation

The investigation began by examining how text is rendered in the PDF pipeline:

1. **HTML/CSS → Layout**: Blitz parses HTML and computes layout using Taffy
2. **Layout → Parley**: Parley performs text shaping, producing glyphs with font metrics
3. **Parley → Krilla**: Fulgur converts Parley glyphs to Krilla glyphs for PDF output

The key file to examine was `crates/fulgur/src/convert/inline_root.rs` in the `extract_paragraph` function, which converts Parley's layout output to Fulgur's internal `ShapedGlyph` format.

### Code Audit

In `inline_root.rs`, the glyph extraction code looked like this:

```rust
for g in glyph_run.glyphs() {
    glyphs.push(ShapedGlyph {
        id: g.id,
        x_advance: g.advance / font_size_parley,
        x_offset: g.x / font_size_parley,
        y_offset: g.y / font_size_parley,
        text_range: 0..text_len,  // BUG: All glyphs get the same range!
    });
}
```

The same issue existed in two other locations:
- `crates/fulgur/src/convert/list_marker.rs` - list marker text
- `crates/fulgur/src/convert/mod.rs` - multi-column paragraph text

### Understanding the Data Model

The debugging process involved understanding the relationship between:

| Entity | Description |
|--------|-------------|
| `Glyph` | A single rendered character from a font |
| `Cluster` | A group of glyphs that represent a single text unit (e.g., a grapheme cluster, or a ligature) |
| `text_range` | The span of text in the original string that this cluster represents |

Key insights:
1. Parley's `GlyphRun` contains multiple `Cluster`s
2. Each `Cluster` has a `text_range()` method that returns the correct text span
3. A `Cluster` can contain multiple `Glyph`s (e.g., for ligatures like "fi")
4. Each glyph within a cluster should get a portion of the cluster's text range

### Krilla API Requirements

The Krilla library's `KrillaGlyph` documentation states:

> `text_range`: The range in the original text that corresponds to the **cluster** of the glyph.

This clarified that the text range should be at the cluster level, not the glyph level. When multiple glyphs belong to the same cluster, the cluster's text range needs to be distributed among them.

## The Fix

### Core Idea

Iterate over **clusters** instead of individual glyphs, use each cluster's `text_range()`, and distribute that range among the glyphs in the cluster.

### Implementation

A helper function was added to `crates/fulgur/src/paragraph.rs`:

```rust
/// Compute the text range for a glyph within its cluster.
///
/// When a cluster has multiple glyphs (e.g., ligatures), we distribute
/// the cluster's text range evenly among them. For single-glyph clusters,
/// the glyph gets the full cluster text range.
pub(super) fn compute_glyph_text_range(
    cluster_range: std::ops::Range<usize>,
    cluster_glyph_count: usize,
    glyph_index: usize,
) -> std::ops::Range<usize> {
    if cluster_glyph_count == 1 {
        // Single glyph cluster - gets the full range
        cluster_range
    } else {
        // Multiple glyphs - distribute the range evenly
        let len = cluster_range.end - cluster_range.start;
        let chunk_size = len / cluster_glyph_count;
        let remainder = len % cluster_glyph_count;

        // Distribute remainder among first 'remainder' glyphs
        let start_offset = chunk_size * glyph_index + (glyph_index).min(remainder);
        let chunk_size = chunk_size + (glyph_index < remainder) as usize;

        cluster_range.start + start_offset..cluster_range.start + start_offset + chunk_size
    }
}
```

## Verification

### Before the Fix

- Selecting any character copied the entire paragraph
- Text selection was effectively broken

### After the Fix

- Text selection works correctly
- Selecting "Hello" in "Hello World" copies only "Hello"
- The PDF's ToUnicode CMap properly maps CID codes to Unicode characters
- Each glyph's text range accurately reflects its position in the original text

## Important Notes

### Why This Works

The fix ensures that:
1. Each **cluster** has the correct text range from Parley
2. Each **glyph** within a cluster gets a portion of that range
3. The PDF reader can map selections to the correct text portions

### Edge Cases Handled

- **Single-glyph clusters**: The glyph gets the full cluster text range
- **Multi-glyph clusters** (ligatures): The cluster's text range is divided among the glyphs
- **Remainder distribution**: If the text length doesn't divide evenly, the first few glyphs get one extra character each

### Limitations

- The fix assumes that text is evenly distributed among glyphs in a cluster. For complex scripts where this assumption may not hold, further refinement might be needed.
- The fix doesn't affect the visual appearance of the PDF, only the text extraction/selection behavior.

## Test Fallout and Resolution

### The Tests That Broke

After applying the fix, 8 tests in `crates/fulgur/tests/render_smoke.rs` failed against the same buggy assumption. Two more tests written in the same style landed from upstream in `fix(gcpm): out-specify surviving author rules in CounterPass injection` (PR #434) while this branch was in flight and were rewritten together with the original eight — 10 tests total:

- `pseudo_multi_item_content_renders_all_items`
- `pseudo_three_item_counter_content_before_and_single_item_unchanged`
- `pseudo_multi_item_content_via_inline_style_id_selector` (from PR #434)
- `target_counter_in_toc_renders_page_number`
- `target_counter_in_link_loaded_css_triggers_pass_two`
- `target_text_before_resolves_attr_pseudo`
- `target_text_empty_for_missing_pseudo`
- `target_text_first_letter_typographic`
- `target_text_in_top_center_resolves_via_implicit_href`
- `target_text_after_resolves_counter_via_counter_pass` (upgraded to a `[2]` text-layer assertion in PR #434)

### Why They Broke

These tests decompressed all PDF byte streams and searched for a **contiguous UTF-16BE hex sequence** — for example, `005B0078005D` for the string `[x]`. That assertion relied on an accidental side-effect of the bug.

Krilla has two ways to encode text in a PDF:

1. **ActualText** — when multiple glyphs share the *same* `text_range`, Krilla groups them and writes the text as a single contiguous UTF-16BE hex string in the PDF structure tree, e.g. `/ActualText <FEFF005B0078005D>`.
2. **CMap (ToUnicode)** — when glyphs have *different* ranges (the correct case), Krilla emits CID-encoded `TJ` operators and relies on the font's ToUnicode CMap to map each CID back to a Unicode codepoint. CIDs are assigned in font-subset order, **not document order**, so the rendered text no longer appears as a contiguous byte sequence anywhere in the PDF.

The old buggy code (`text_range = 0..text_len` for every glyph) accidentally triggered path 1: all glyphs sharing the same range caused Krilla to bundle the entire paragraph into one ActualText block, and the rendered string sat as plain UTF-16BE hex inside it. The tests grepped that hex.

The fix correctly triggers path 2 (CMap), which is the right encoding per the Krilla API and the PDF spec. The features under test still work — the characters are rendered in the PDF — but you can't recover them by grepping bytes anymore. You have to walk each `TJ` operator's CIDs and look them up in the font's ToUnicode CMap.

### How the Tests Were Fixed

A `pdftotext`-based extraction helper was added to `render_smoke.rs`:

```rust
fn extract_pdf_text(pdf: &[u8]) -> Option<String> {
    // shells out to `pdftotext -raw` (poppler-utils)
    // returns None if pdftotext is not installed → caller skips
}
```

`pdftotext` walks the ToUnicode CMap to reconstruct the actual rendered text, which is exactly what these tests need. The `-raw` flag flattens tagged PDFs to reading-order content stream text and avoids the column wrap that pdftotext applies in default mode (which would split sentinels like `[APP: ]` across line breaks).

`poppler-utils` is already a CI dependency (`.github/workflows/ci.yml` installs it for `fulgur-wpt` reftests), and one existing test (`target_text_second_arg_resolves_target_fragments`) was already using `pdftotext` with the same graceful-skip pattern. No new dependencies were added.

Each test was rewritten to assert against the extracted string with `text.contains(...)`, preserving the original contiguity / adjacency semantics:

| Test | Assertion |
|------|-----------|
| `target_counter_in_toc_renders_page_number` | `contains("(p.2)")` and `contains("(p.3)")` |
| `target_counter_in_link_loaded_css_triggers_pass_two` | `contains("(p.2)")` and `contains("(p.3)")` |
| `target_text_in_top_center_resolves_via_implicit_href` | `contains("Header: My Section Title")` |
| `target_text_before_resolves_attr_pseudo` | `contains("[APP: ]")` — preserves the trailing-space invariant |
| `target_text_empty_for_missing_pseudo` | `contains("[]")` — preserves the adjacency invariant |
| `pseudo_multi_item_content_renders_all_items` | `contains("[x]")` |
| `pseudo_three_item_counter_*` | `contains("[1]")` and `contains("Body.")` |
| `target_text_after_resolves_counter_via_counter_pass` | `contains("[2]")` |
| `pseudo_multi_item_content_via_inline_style_id_selector` | `contains("[2]")` |

#### `target_text_first_letter_typographic` — added the CJK font to the test fixture

The HTML keeps `「Hello」` (U+300C / U+300D CJK corner brackets) — the canonical CSS Pseudo-Elements 4 §3.2 case. But the asset bundle from `tagged_render_with_noto` only carried `NotoSans-Regular.ttf`, which doesn't contain `「`. Without a glyph, Stylo silently fell back to a system font — whichever happened to be on the test runner — and the corner bracket landed in the PDF as CID 0 of that fallback with no usable ToUnicode CMap entry. So how did the test ever pass?

Because the bug's ActualText path bypassed the CMap entirely. Krilla's `/ActualText <FEFF...>` marker is written from the *source text* Krilla was handed, not derived from the font's ToUnicode tables. With `text_range = 0..text_len` on every glyph, Krilla wrote one big ActualText payload containing `300C0048…300D` verbatim. The test grepped for `300C0048` and hit. The visible glyph for `「` was system-dependent (and possibly a missing-glyph box), but the assertion didn't care — it was effectively checking that the *source* string passed to Krilla contained `「H`, not that the rendered PDF could be text-extracted as `「H`.

Once the fix routes character recovery through the ToUnicode CMap (the spec-correct path), that side channel disappears and the rendering font has to actually carry `「`. Layer `NotoSansJP-Regular.otf` (already bundled in `examples/.fonts/` for the JP examples and Chrome VRT) into the asset bundle for this test alongside the Latin Noto Sans, and set `font-family: 'Noto Sans JP', 'Noto Sans', sans-serif` so Stylo picks the JP face for the CJK punctuation and Noto Sans for the rest. With a real ToUnicode entry behind `「`, pdftotext recovers `「H` and the assertion runs `text.contains("「H")`. This also incidentally makes the test's visual rendering deterministic — previously it relied on a system fallback.

The pure-Rust algorithm — including its CJK punctuation handling — is independently covered by unit tests in `crates/fulgur/src/gcpm/target_ref.rs` (`first_letter_skips_leading_punct_and_keeps_trailing` exercises `「『Hello』」 → 「『H`). The render_smoke test verifies the end-to-end render plumbing, not the algorithm.
